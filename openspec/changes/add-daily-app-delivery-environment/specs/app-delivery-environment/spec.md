## ADDED Requirements
### Requirement: Daily app scaffolding
The platform SHALL provide a one-command scaffold that creates a new repo with a Docker/Docker Compose frontend+backend template, health endpoint, environment-based configuration, example tests, and GitHub Actions CI/CD config for Cloudflare Pages and AWS ECS.

#### Scenario: New app scaffolded
- **WHEN** a developer runs the scaffold command with an app name
- **THEN** a new repo is created with the template, docker-compose file, CI/CD workflow file, and instructions to push to the shared registry

### Requirement: AI-first development environment
The platform SHALL provide a Docker-based dev container per repo with required CLIs (Node, package manager, git, AWS/Cloudflare/GitHub CLIs) and git worktree helpers to enable parallel feature development and local testing.

#### Scenario: Worktree-based development
- **WHEN** a developer (or AI agent) opens the repo in the dev container
- **THEN** they can create git worktrees for multiple features, run docker-compose locally with the baked-in tooling, and execute tests without additional setup

### Requirement: One-command CI/CD deployment
The platform SHALL ship a default GitHub Actions workflow that, on push to the main branch, builds and tests, builds/pushes backend images, deploys to ECS, builds/uploads frontend artifacts to Cloudflare Pages, performs a smoke test, and awaits manual approval before production promotion.

#### Scenario: Main branch deployment
- **WHEN** code is pushed to the main branch of a scaffolded app
- **THEN** the workflow builds and tests, publishes the backend image, deploys to ECS, uploads the frontend to Cloudflare Pages, runs a smoke test, and pauses for manual production approval

### Requirement: Frontend delivery via Cloudflare Pages
The platform SHALL provide a Cloudflare Pages project per app with an autogenerated subdomain and TLS certificate, and publish build artifacts from CI/CD without manual steps.

#### Scenario: Frontend published
- **WHEN** CI/CD completes a successful build of the frontend
- **THEN** the artifacts are uploaded to the app’s Cloudflare Pages project and served via HTTPS at the assigned subdomain

### Requirement: Backend delivery via AWS ECS
The platform SHALL deploy backend containers to AWS ECS (Fargate by default) with HTTPS routing, per-app secrets, logging/metrics sinks, and resource limits provisioned by IaC.

#### Scenario: Backend deployed
- **WHEN** CI/CD publishes a backend image and runs deployment
- **THEN** an ECS service is updated with the new task definition, traffic is routed over HTTPS, secrets are mounted, and logs/metrics flow to the shared observability stack

### Requirement: Rollback and disablement
The platform SHALL provide an automated rollback or disablement path when a smoke test fails or a deployment is manually rejected.

#### Scenario: Failed deployment rollback
- **WHEN** the post-deploy smoke test fails or the manual approval is denied
- **THEN** the workflow rolls back to the previous stable release or disables traffic routing for the app slot

### Requirement: Issue-driven AI automation
The platform SHALL offer GitHub issue templates and automation that trigger Claude to implement tasks and Codex to refine requirements/specs, with guardrails on allowed commands/paths and auditability of actions.

#### Scenario: Issue triggers implementation
- **WHEN** an issue is created using the implementation template
- **THEN** Claude is triggered to execute the defined steps within the allowed scope, producing code changes and status updates while respecting guardrails
